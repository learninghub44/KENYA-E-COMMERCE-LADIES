# Contributing to Zuri Market

Thank you for your interest in Zuri Market. Please read this guide
before opening an issue or pull request.

## Before You Contribute

Zuri Market is proprietary software (see [`LICENSE`](./LICENSE) and
[`TERMS_OF_USE.md`](./TERMS_OF_USE.md)). Access to this repository does
not, by itself, grant any right to contribute code, and submission of a
pull request does not create any license or ownership claim over the
Software.

External contributions are only accepted from individuals or
organizations with a written contributor agreement in place with the
Owner. If you are interested in contributing and do not already have
such an agreement, contact the Owner (see [`OWNER.md`](./OWNER.md))
before submitting work.

## How to Report Bugs or Request Features

You do not need a contributor agreement to report bugs or request
features — see [`SUPPORT.md`](./SUPPORT.md) for how to reach us. Use the
issue templates under `.github/ISSUE_TEMPLATE/` so reports include the
information needed to reproduce and triage them.

## Contribution Workflow (Authorized Contributors)

1. Confirm you have a written contributor agreement with the Owner.
2. Discuss significant changes in an issue before starting work, so
   direction and scope are agreed upon in advance.
3. Fork or branch according to the access you've been granted.
4. Write clear, atomic commits. Follow the existing commit message
   conventions in this repository (e.g. `feat:`, `fix:`, `docs:`,
   `legal:`).
5. Ensure all tests pass and add coverage for new behavior.
6. Open a pull request using the template in
   `.github/PULL_REQUEST_TEMPLATE.md`, describing what changed and why.
7. Be responsive to review feedback. The Owner (or designated
   maintainer) has final approval authority on all merges — see
   [`GOVERNANCE.md`](./GOVERNANCE.md).

## Code Standards

- Match the existing code style and conventions used in the relevant
  part of the codebase.
- Keep pull requests focused — one logical change per PR.
- Include tests for new functionality and bug fixes where practical.
- Update relevant documentation alongside code changes.
