# Primary Key Strategy

## Standard

Use UUID primary keys generated by Postgres for application-owned entity tables.

```sql
id uuid primary key default gen_random_uuid()
```

This keeps identifiers safe to expose through APIs and stable across future service boundaries, imports, previews, and mobile clients.

## Supabase Auth Alignment

`profiles.id` references `auth.users.id` directly and does not generate its own UUID. The auth user ID is the profile ID.

## Composite Keys

Use composite primary keys for pure relationship tables when no independent lifecycle is needed.

Examples:

- `user_roles (user_id, role)`
- `seller_members (seller_id, user_id)`
- `wishlist_items (wishlist_id, product_id)`
- `collection_products (collection_id, product_id)`
- `store_followers (seller_id, user_id)`

## Human-Readable Identifiers

Human-readable identifiers are not primary keys.

Examples:

- `orders.order_number` is unique for customer support and receipts.
- `products.slug` is unique per seller for storefront URLs.
- `sellers.slug` is globally unique for storefront URLs.
- `coupons.code` is unique for redemption lookup.

## Anti-Patterns

- Do not use incremental integer IDs for public resources.
- Do not use slugs as primary keys.
- Do not reuse third-party provider IDs as primary keys.
- Do not use compound natural keys for entities with future workflow state.
